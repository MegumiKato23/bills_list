# BookKeeper 账单列表技术方案

## 1. 项目背景

实现 Android 端账单列表，核心目标不是完整记账闭环，而是验证大数据量账单在 Flutter 中的连续浏览能力。项目需要支持 1 万到 10 万级账单数据，保证首屏尽快可见、滑动过程无明显等待、弱网下仍可查看缓存数据，并避免 UI 层内存随总账单数线性增长。

当前实现采用“Flutter 懒渲染列表 + Riverpod 状态控制 + SQLite 本地缓存 + 远端游标分页接口”的方案。

## 2. 设计目标

1. 首屏只加载必要数据，有缓存时优先显示缓存，无缓存时只请求第一页。
2. 列表使用懒渲染，不一次性构建所有账单 Widget。
3. 分页使用稳定游标，不使用 offset 做远端深分页。
4. 页面层只持有有限窗口数据，长列表连续滑动时内存趋于稳定。
5. 弱网、离线、分页失败时不清空已有列表。
6. 日期相同的账单聚合展示，并统计当天收入、支出。
7. 支持 fake 数据源生成 10 万条账单，用于滑动和弱网压测。

## 3. 总体架构

```text
Flutter UI
  -> BillListPage
  -> BillListController
  -> BillRepository
  -> BillLocalDataSource / BillRemoteDataSource
  -> Drift SQLite / Dio API / FakeBillDataSource
```

### 3.1 UI 层

UI 层由 `BillListPage`、`BillDateHeader`、`BillRow` 组成。

- `CustomScrollView + SliverList` 负责按需构建可见区域。
- `RawScrollbar` 显示右侧动态滑块，内容越长滑块越短。
- `RefreshIndicator` 支持下拉刷新。
- `BillRow` 拆成独立 Widget，并用账单 id 作为稳定 key。
- 日期头和账单行通过 `BillListDisplayEntry` 统一描述，避免页面直接处理分组逻辑。

### 3.2 状态层

状态层由 `BillListController` 管理，基于 Riverpod Notifier。

- `initialize()`：先读取本地缓存窗口，再后台刷新远端第一页。
- `refresh()`：刷新第一页，成功后重新读取本地窗口。
- `loadMore()`：接近底部时自动请求下一页。
- `_requestEpoch`：刷新时递增版本号，忽略旧分页结果，避免异步请求覆盖新状态。
- `isLoadingMore`：保证同一时间只有一个分页请求。
- `windowOffset`：记录当前 UI 窗口在本地缓存中的偏移。
- `_backfillIfNeeded()`：向上滑回历史区域时从 SQLite 回补窗口数据。

### 3.3 数据层

数据层由 `BillRepositoryImpl` 对 UI 隐藏“远端 + 本地缓存”的细节。

- `readWindow()` 只读 SQLite 的一个窗口。
- `refreshFirstPage()` 请求远端第一页，成功后 upsert 到本地。
- `loadMore()` 请求远端下一页，成功后 upsert 到本地。
- 网络错误时返回 `SyncResult.failure`，页面保留已有缓存数据。
- fake 数据源首页刷新会清理旧 fake 缓存，避免测试数据规则变化后新旧分布混合。

### 3.4 本地缓存

本地缓存使用 Drift + SQLite。

- `bills.id` 是主键，重复接口数据通过 upsert 去重。
- 排序字段为 `occurredAtMillis DESC, id DESC`。
- 建立索引 `idx_bills_occurred_id`，让窗口查询由 SQLite 完成排序。
- 金额以 `amountCents` 整数保存，避免浮点金额误差。

## 4. 核心流程

### 4.1 首屏加载

```text
进入页面
  -> Controller.initialize()
  -> SQLite readWindow(limit: 800, offset: 0)
  -> 立即显示缓存
  -> 后台 refreshFirstPage(pageSize: 80)
  -> 远端成功后 upsert SQLite
  -> 重新读取窗口并刷新 UI
```

这样有缓存时首屏不依赖网络；无缓存时只显示加载态并请求第一页。

### 4.2 自动分页

```text
用户滑动
  -> 监听 ScrollController
  -> 剩余距离 < 2.5 屏
  -> loadMore()
  -> 远端按 cursor 请求下一页
  -> upsert SQLite
  -> 从 SQLite 读取 UI 窗口
```

分页成功后只追加本地缓存，不把所有历史数据长期留在 Flutter 状态中。

### 4.3 UI 窗口控制

当前常量：

| 指标                   | 当前值 | 说明                    |
| ---------------------- | -----: | ----------------------- |
| `pageSize`             |     80 | 每页请求数量            |
| `uiWindowMax`          |    800 | UI 层最多常驻账单条数   |
| `uiWindowBackfillStep` |    240 | 向上回补窗口步长        |
| `preloadScreenFactor`  |    2.5 | 距底部 2.5 屏触发预加载 |
| `billRowExtent`        |     88 | 固定账单行高度          |
| `billDateHeaderExtent` |     56 | 日期头固定高度          |

当当前窗口达到 800 条后，继续向下分页会推进 `windowOffset`，释放窗口前端数据对象。用户向上滑回时，再从 SQLite 回补前面的窗口。

### 4.4 日期合并

日期合并由 `buildBillListDisplayEntries()` 完成。

- 按账单发生时间的本地日期生成日期分组。
- 每个日期组生成一个 `BillDateHeaderEntry`。
- 日期头统计当天收入、支出。
- 账单行标记 `single / first / middle / last`，用于控制圆角、边框和组内贴合效果。

页面只负责渲染 display entry，不在 `build()` 里做排序、接口请求或复杂计算。

## 5. 优化前后技术指标对比

下表的“优化前”指普通全量加载列表方案或项目早期未做分页窗口控制的实现；“优化后”指当前代码方案。DevTools 帧耗时和内存峰值仍需要在 Profile 模式下实测补齐。

| 指标           | 优化前                    | 优化后                                   | 变化                        |
| -------------- | ------------------------- | ---------------------------------------- | --------------------------- |
| 首屏远端数据量 | 可能一次拉取全部 10 万条  | 只请求第一页 80 条                       | 首屏请求量降低约 99.92%     |
| UI 层常驻数据  | 随总数增长，最多 10 万条  | 默认最多 800 条窗口数据                  | UI 数据对象上限降低约 99.2% |
| 列表构建方式   | 可能一次性构建大量 Widget | `SliverList` 懒构建可见区域              | 降低首屏构建压力            |
| 深分页方式     | offset 或全量排序后截取   | cursor 分页，按发生时间和 id 稳定排序    | 避免深分页漂移和重复        |
| 本地排序       | Dart 数组全量排序风险高   | SQLite 索引排序后窗口查询                | 减少 Dart 主线程计算        |
| 缓存策略       | 网络失败可能无数据显示    | 先读 SQLite，失败保留缓存                | 弱网可用性提升              |
| 重复数据处理   | append 容易重复           | SQLite 主键 upsert 去重                  | 数据一致性提升              |
| 加载更多防重   | 快速滑动可能重复请求      | `isLoadingMore` 防重                     | 降低重复分页请求            |
| 刷新与分页并发 | 旧分页可能覆盖刷新结果    | `_requestEpoch` 忽略旧结果               | 避免状态回退                |
| 分页失败处理   | 可能清空列表或全屏错误    | 保留列表，底部显示重试                   | 用户浏览不中断              |
| 预加载体验     | 到底后才加载              | 距底部 2.5 屏预加载                      | 降低等待感                  |
| 行高稳定性     | 高度不稳定，滚动估算差    | 固定行高 88                              | 滚动窗口计算更稳定          |
| 日期聚合       | 逐条平铺                  | 按日期头合并，并显示收支统计             | 信息层级更清晰              |
| 测试覆盖       | 无明确保障                | `flutter analyze`、`flutter test` 已通过 | 基础质量可回归              |

## 6. 预期性能收益

### 6.1 首屏性能

优化后首屏只需要处理第一页请求和有限本地窗口读取。有缓存时，页面可以先展示 SQLite 中的历史数据，再后台刷新远端第一页，避免网络等待直接阻塞首屏。

### 6.2 滚动性能

列表使用 `SliverList` 懒渲染，可见区域外 Widget 不会全部构建。固定行高、有限窗口和预加载阈值配合后，快速滑动时可以减少临时对象创建和分页等待。

### 6.3 内存稳定性

账单总量从 1 万增长到 10 万时，SQLite 可以继续保存全部缓存，但 UI 状态默认只保留 800 条账单窗口。深度滚动后旧窗口数据会被释放，内存增长不应随总账单数线性增加。

### 6.4 弱网稳定性

本地缓存和远端同步分离后，网络失败不会影响已缓存列表。刷新失败时展示离线 Banner，分页失败时仅在底部展示重试入口。
